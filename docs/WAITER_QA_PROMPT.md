# Waiter Module — QA & Bug-Hunt Brief

You are a senior Flutter engineer + QA lead. Your job is to **audit the Waiter Module of the Hermosa POS app, find bugs, and produce a test plan**. This is an analysis pass: **do not modify any code, do not write test files** — produce the two Markdown deliverables described at the end.

## 1. Scope

In scope:
- Everything under `lib/waiter_module/**` (services, screens, dialogs, widgets, models, `waiter_module_entry.dart`).
- Its integration seams: `lib/locator.dart` (getIt registrations), `lib/screens/login_screen.dart` + `lib/screens/branch_selection_screen.dart` (how `WaiterModuleEntry` launches), `lib/screens/table_management_screen.dart` (the cashier "viewer" hookup), and the cashier mesh bootstrap.
- Shared services the module *calls*: `DisplayAppService` (KDS), `OrderService`, `BranchService`, `AuthService`, `ProductService`, `TableService`, the NearPay stack, `ReceiptBuilderService`/`InvoicePrintWidget` — audit how the waiter module *uses* them, not their internals.

Out of scope: the salon module, the restaurant cashier order flow internals, anything unrelated to the waiter feature.

## 2. The spec (your oracle)

`docs/SRS_Waiter_Module.md` is the reverse-engineered spec — requirement IDs `FR-<AREA>-<n>` across these areas: SES (session/shift), DISC (discovery/roster), NET (mesh transport + auth), CTL (controller coordination), TBL (table floor plan/registry), ORD (order composition/KDS/outbox), BILL (billing/payment/receipt), PU (pickup hand-off "استلام"), MIG (table migration), WL (waitlist), MSG (notifications/calls/chat), CFG (config sync cashier→waiter), PRN (per-device printer settings).

Treat the SRS as the statement of intended behavior. **Where the code disagrees with the SRS, that is itself a finding** — report it (and say which one you think is right).

## 3. Hard constraints — do not violate

- **Do not change any code.** No edits, no refactors, no "while I'm here" fixes. Output is documents only.
- **Do not write or modify test files** either — describe test cases in prose, don't author `*_test.dart`.
- Printing/receipt code (`ReceiptBuilderService`, `InvoicePrintWidget`, the print dispatcher's interaction with the print stack) is **off-limits to modify**; you may report bugs in how the waiter module *invokes* it, but flag, don't fix.
- The waiter UI is valid only for `branchModule == 'restaurants'` — keep that in mind when reasoning about gating.

## 4. Method

1. **Read, don't skim.** The module is ~41 files / ~13K lines — read the service files *in full* (`waiter_controller.dart`, `waiter_network_service.dart`, `waiter_table_registry.dart`, `waiter_cart_store.dart`, `waiter_order_outbox.dart`, `waiter_billing_service.dart`, `waiter_session_service.dart`, `waiter_discovery_service.dart`, `waiter_roster_service.dart`, `waiter_pickup_store.dart`, `waiter_message_store.dart`, `waiter_config_store.dart`, `waiter_kitchen_bridge.dart`, `waiter_notification_service.dart`, `waiter_print_dispatcher.dart`, `mesh_auth_service.dart`, `waiter_device_prefs.dart`), then the screens (`waiter_order_screen.dart`, `waiter_tables_screen.dart`, `waiter_home_screen.dart`, `waiter_messages_screen.dart`, `waiter_profile_screen.dart`, `waiter_printer_settings_screen.dart`, `waiter_login_screen.dart`), the dialogs, widgets, and models. Use parallel sub-agents for breadth, but verify every claim against the actual source — do not trust an excerpt.
2. **Walk every FR area** in §2's list. For each: confirm the code implements it, then probe the edges.
3. **Trace data flows end-to-end**, not just per-file: a table event from `broadcastTableEvent` → registry apply → self-echo stream → `WaiterTablesScreen` rebuild → `WaiterTableCard` state, *and* over the wire → peer `_handleIncoming` → registry → cashier `table_management_screen.dart`. A KDS order from `WaiterKitchenBridge` → `DisplayAppService` *or* → `WaiterOrderOutbox` → reconnect flush. A bill from `WaiterBillingService.processBill` → `OrderService.createBooking` → NearPay → `WaiterPrintDispatcher` → `ReceiptBuilderService`.
4. **Think adversarially about concurrency & failure**: two waiters acting on the same table; cashier + waiter both closing a table; Wi-Fi drop mid-broadcast; KDS socket down; app force-killed mid-`dispose()` / mid-persist / mid-bill; waiter signs out and a different waiter signs in on the same device ("device hand-off"); cashier restarts while a pickup is in flight; mDNS fails; backend returns 401 to a waiter; clock skew between devices.
5. **Check the role/identity guards** (FR-NET-8, FR-MSG-1, FR-PU-1, FR-MIG-1, FR-CFG-1): can a waiter do something only a viewer should, or vice versa? Are sender==claimer / sender==accepter / `viewer-` prefix checks actually enforced on the receive path, not just the send path?
6. **Check teardown completeness** (FR-SES-10): does `clearSessionStores` actually wipe *everything* listed (messages, pickup store, table registry incl. disk, roster, cart, **outbox**, billing caches, BranchService caches, mesh MAC key) — and does it swallow exceptions in a way that could leave one store un-wiped? Does anything survive that shouldn't (revenue/tip attribution leaking to the next waiter)?

## 5. Known symptom to root-cause first

> **"After a waiter opens a table and then leaves the order screen without sending anything, the table still shows that 'someone is taking an order' (جاري اخذ الطلب) — on the waiter's own tables grid and/or on the cashier's table-management screen."**

This is a real reported bug. Root-cause it thoroughly. Things to trace (verify line numbers yourself; these are leads, not gospel):

- `WaiterController.activeOrderingTableId` / `setActiveOrderingTable` / `clearActiveOrderingTable(tableId?)` — note `clearActiveOrderingTable` returns early if the passed `tableId` doesn't match the current one. (This flag is in-memory and only gates banner sounds — but confirm.)
- `WaiterOrderScreen.initState()` → `_announceAssignment()`: on entry it broadcasts a `TableLifecycleEvent` — `TableLifecycleKind.takingOrder` when the local cart is empty, `assigned` when the waiter already has items. So **merely opening the order screen marks the table "taking order" on every peer.**
- `WaiterOrderScreen.dispose()`: it only broadcasts a `released` event if **all** of these hold — the waiter owns the table (or nobody does), the local cart is empty, the table isn't already `paymentPending`/`paid`, and it "was just taking an order". Enumerate **every** way one of those conditions can be false (e.g. `session.self == null` due to a logout race ⇒ no owner-me ⇒ no release; another waiter claimed the table between init and dispose; the table got bumped to `paymentPending`; an exception earlier in `dispose`; the controller/network already stopped so the broadcast is dropped; the screen popped without `dispose` running its release path).
- App **force-killed during/just after `dispose()`**: the `released` broadcast never goes out, and `WaiterTableRegistry` has by then persisted `takingOrder=true` to disk. On next launch `WaiterTableRegistry.hydrate` reloads it, and `reconcileWithBackend()` has a guard that **keeps self-owned `takingOrder` rows even if the backend says the table is free** — so the stale state can survive forever. Confirm this path and whether anything ever clears it.
- Cashier side: `lib/screens/table_management_screen.dart` keeps a `_takingOrderTables` set, populated on a `takingOrder` event and removed only on `assigned` / `updated` / `paymentPending` / `paid` / `released`. If `released` never arrives (any of the above), the cashier card shows "جاري اخذ الطلب" indefinitely. Also check whether the cashier ever ages out / reconciles that set.
- Also consider the *other* entry points that can announce a table without a matching release: opening then dismissing `table_details_dialog.dart`; `UPDATE_CART` live-preview pushes (`FR-ORD-4`) that re-assert occupancy after a `released`; migration leaving the old table marked; an `ORDER_EDIT`/`ORDER_CANCEL` that doesn't update the registry.

Deliver: a numbered list of **every distinct code path** that can leave a table stuck in `takingOrder`, each with file:line, the precondition that triggers it, and a one-line fix sketch (sketch only — do not implement).

## 6. Bug-hunt checklist (work through all of it)

Go FR-area by FR-area. For *every* bug/risk you find, record: **title · severity (Critical/High/Medium/Low) · file:line · what's wrong · how to reproduce · root cause · suggested fix (sketch only)**.

Specific things to verify (these are hypotheses surfaced by a quick first pass — confirm or refute each against the source; some may be non-issues):

- **Session/teardown:** Does `clearSessionStores` await the outbox wipe before a new sign-in can race it? Is there any async flow (`WaiterBillingService.processBill`, `WaiterOrderOutbox.flushIfConnected`, `WaiterPrintDispatcher`) with **no session-generation guard**, so it can complete under a *different* waiter after a hand-off? Does `signOut`'s "write empty string then remove" leave a half-state on a force-kill?
- **Persistence atomicity:** `WaiterTableRegistry` and `WaiterCartStore` claim a dual-slot backup + a `_persistNext` future chain — does that chain actually exist and serialize correctly? Is anything important persisted only on a 300ms debounce (so a force-kill in that window loses it)? Does the cart get an immediate flush after "create booking" succeeds, or can a crash there leave stale drafts?
- **Mesh transport:** Is the inbound-HELLO 5s timeout actually wired? Does the 512-id dedup LRU evict correctly, and what happens at >512 messages in the window (duplicate kitchen tickets? double pickup-ack?)? Does the outbound reconnect have any backoff cap, or does it spin every 3s forever for an unreachable peer until the sweep forgets it? Does `closeConnectionTo` actually drop *and forget the address* so a stale post-branch-switch IP can't reconnect?
- **Mesh auth:** Is the HMAC verified **before** parse/dedup/handle? Constant-time compare? Is the key cleared on logout *and* branch switch? Does the "accept unsigned during pre-login boot" window ever stay open after the key is hydrated (would let a LAN attacker in)?
- **Roster/discovery:** Branch filter applied at discovery time *and* on the envelope? Does `id.substring(0,6)` blow up for a UUID shorter than 6 chars? Does `lastSeen` refresh on *any* inbound message (not just heartbeats)? Stale threshold (45s) vs heartbeat (15s) vs sweep (10s) — any window where a live peer flickers offline?
- **Table registry state machine:** `apply()` for each of `takingOrder`/`assigned`/`released`/`updated`/`paymentPending`/`paid` — does `assigned` correctly clear `takingOrder`? Does `released` fully delete the row (incl. `orderId`)? Is the out-of-order guard (drop `paymentPending` if already `paid` for the same `orderId`) correct, and does it handle a *different* `orderId`? Does `hydrate` rewriting `waiterId` to the current session ever mis-attribute a row?
- **Order/cart/outbox:** Does the `NEW_ORDER` envelope the waiter sends byte-match the cashier's shape (so the KDS needs no changes)? Does sending an order also broadcast the matching `TABLE_UPDATE`? Outbox: single-writer mutex between enqueue and flush honored? Progress persisted after *each* send (crash-safe)? Mid-flush KDS-drop aborts cleanly and keeps the rest? `_retries` cap (10) actually drops the poison entry? Is the outbox wiped on signout? Can two waiters' orders ever interleave under one identity?
- **Billing/payment/receipt:** `round2(subtotal × (1+rate))` with `taxAmount = round2(subtotal × rate)` — does this match what the cashier computes for the same cart (same `digitsNumber` rounding)? On a NearPay decline after the booking was created, does `WaiterBillResult` carry `bookingId`/`invoiceId` so the retry doesn't double-book? Pay-methods/tax caches refreshed when the order screen opens and cleared on signout? Does the receipt go through `ReceiptBuilderService`/`InvoicePrintWidget` unchanged (FR-BILL-6)?
- **Pickup ("استلام"):** Only `viewer-`-prefixed senders can issue/cancel a pickup (enforced on receive)? First claim wins, later claims dropped locally, UI never reverts to an older claimer? Claiming folds in a `TABLE_ASSIGN` so the cashier flips to "occupied by X"? Orphan-claim recovery (cashier restarted) synthesizes a request record? On HELLO each waiter replays its own claimed pickups? Is the pickup banner sound suppressed while that waiter is composing an order — but the request still persisted so it surfaces afterward (FR-CTL-7)?
- **Migration:** Only the cashier or the *owning* waiter can migrate? `old==new` is a no-op? The owner moves the cart (drafts+sent+guests), releases the old table, assigns the new one carrying `guestCount`/`total`/`itemCount`/`items` **and the pay-later `orderId`** (so "Create Invoice" on the moved table invoices the original booking, not a new one)? If the old table was `paymentPending`, is that re-established on the new id? Do non-owner peers correctly just observe?
- **Waitlist:** Every local mutation rides the mesh as a `WAITLIST_EVENT` with a self-echo on the local stream? New device gets a `WAITLIST_SNAPSHOT` catch-up, reconciled last-write-wins? Any way the snapshot clobbers a newer local change?
- **Notifications/calls/chat:** Waiter→waiter calls hard-guarded (StateError, viewer-only send)? A receiving waiter surfaces a message only if it's a broadcast (and receiver isn't a viewer) or addressed to it? First accepter broadcasts `WAITER_CALL_ACCEPTED`, others flip to "تم الاستلام بواسطة X" and hide Accept? Unread badge correct; mark-read owned by `WaiterHomeScreen` not the screen's `initState`?
- **Config sync (cashier→waiter):** Only a viewer produces authoritative `CONFIG_KITCHEN_PRINTERS` / `CONFIG_KDS_ENDPOINT`; waiters never do? Version-gated/idempotent? Persisted so a cold-start waiter has them before the first `NEW_ORDER`? On signed-in resume, is the stored KDS endpoint re-applied to the live `DisplayAppService`?
- **Printer settings:** The waiter printer hub reuses the cashier's `PrintersTabView` and the device/category/kitchen-route registries verbatim (no fork)? Language tab actually drives invoice language?
- **Lifecycle/wiring:** getIt registrations in `lib/locator.dart` — dependency order correct, all lazy singletons, nothing constructed twice? `WaiterModuleEntry` boot order matches FR-SES-1 (hydrate session → config store → outbox → *then* mesh)? `start()` single-flight and unwinds cleanly on a mid-flight `stop()` / session bump (FR-CTL-1/2)? Streams guarded against emit-after-dispose?
- **General Dart hazards:** un-awaited futures whose failures vanish; `StreamController`s not closed; `Timer`s/`subscription`s not cancelled in `dispose`; `setState` after `dispose`; `BuildContext` used across an `await`; null-bang `!` on values that can be null in a race; `==`/`hashCode` mismatches on model classes used as map keys; integer/double rounding drift; locale-dependent string parsing of TXT records or amounts.

## 7. Test scenarios to write

Produce a structured **manual + integration test plan** (prose test cases, not code). Organize by FR area. Each test case: **ID · title · preconditions/setup · steps · expected result · which FR(s) it covers · priority**. Cover at minimum:

- **Setup matrices:** 1 waiter alone; 1 waiter + cashier(viewer); 2+ waiters + cashier; restaurant branch with NearPay on vs off; Android/Sunmi vs iPad (note NFC/NearPay differences).
- **Session/shift:** cold start auto-sign-in from profile; name fallback chain; retry splash when no name resolves; end shift broadcasts `WAITER_LEAVE` and wipes session stores; **device hand-off** — waiter A ends shift, waiter B signs in on the same tablet, B must see *none* of A's tables/drafts/queued orders/notifications; branch switch (cashier) re-advertises new branch id; availability status changes broadcast.
- **Discovery/roster:** two devices find each other; peer on a different branch is ignored; peer goes silent → marked offline after ~45s; offline peer comes back and reconnects; mDNS unavailable → "alone on LAN", no crash.
- **Mesh/auth:** rogue LAN device sending unsigned/forged messages is ignored; cross-branch replay rejected; protocol-version-2 message dropped; duplicate message (WS flap) doesn't double-fire (no double kitchen ticket, no phantom payment-pending, no double pickup-ack); pre-login boot mesh still works before keys hydrate.
- **Table floor plan:** open a table → it shows "taking order" on peers; **leave the order screen without sending → it returns to free on the waiter grid AND the cashier screen** (the reported bug — make this an explicit, prominent test, including the force-kill-during-exit and restart variant); send an order → table fills (item count/total/owner) everywhere; lone-waiter table state survives an app restart.
- **Order/KDS/outbox:** compose with customizations + guest count; send to KDS while online (KDS shows it, same shape as cashier); send while KDS down → enqueued; reconnect → auto-flush, in order, no duplicates; app killed mid-compose → drafts restored; app killed mid-flush → no duplicate sends; poison order dropped after 10 retries; outbox empty after signout.
- **Billing/payment/receipt:** close a table cash → booking+invoice created, receipt prints matching the cashier receipt byte-for-byte (seller block, tax number, CR, logo, daily order number); close with NearPay card → in-app flow → success; NearPay decline after booking created → retry without double-booking; pay-later booking → table goes `paymentPending`; backend pay-later booking with no local trace → recovered by start-time reconcile; send invoice via WhatsApp.
- **Pickup:** cashier requests pickup → all waiters alerted (sound + banner), cashier doesn't sound; first waiter to accept claims it, cashier flips to "occupied by X", other waiters' banners clear; second claimer is dropped, UI doesn't revert; cashier cancels while pending → cleared; cashier cancels after claimed → no-op; cashier restarts mid-claim → orphan-claim recovery; waiter composing an order doesn't hear the pickup sound but sees it after closing the order screen.
- **Migration:** cashier migrates an occupied table → cart/owner move, old table frees; owning waiter migrates their own table → same; non-owner waiter tries to migrate → rejected; migrate a `paymentPending` table → new table is `paymentPending` and "Create Invoice" invoices the original booking; `old==new` → no-op.
- **Waitlist:** add/update/remove/notify/seat/cancel a party on one device → mirrored on all; new device joins → gets full snapshot; concurrent edits on two devices → converge; notify via WhatsApp.
- **Notifications/calls:** cashier calls a waiter (directed) → only that waiter alerted; cashier broadcasts "call a waiter" → all alerted; first accepter → others see "accepted by X", Accept hidden; waiter→waiter call attempt → blocked; unread badge correct; mark-read on tab switch.
- **Config sync:** waiter joins → cashier pushes printer list + KDS endpoint on HELLO; cashier changes a kitchen printer → pushed to all waiters; repeated pushes idempotent; cold-start waiter has config before first `NEW_ORDER`; signed-in resume re-applies the KDS endpoint.
- **Printer settings:** waiter adds/edits/tests/removes a printer via the shared UI; sets primary/secondary invoice language → reflected on the next receipt.
- **Resilience:** Wi-Fi drops for 30s mid-shift then recovers — peers reconnect, no data loss; backend 401 to a waiter doesn't tear down the session; partial `start()` failure (e.g. port bind fails) doesn't wedge the controller; force-kill at each of: mid-compose, mid-send, mid-flush, mid-bill, mid-`dispose()`, mid-persist — and verify clean recovery on restart.
